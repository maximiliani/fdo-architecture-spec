Overview

The vision of the FDO concept is to enable machine actionability and facilitate highly automated processing of data. If an FDO client finds a PID that identifies an FDO, it is able to automatically interpret and process the FDO, regardless of the technology used to implement it. The basis for enabling automated processing regardless of the used technology is that FDO implementations comply with the rules specified in this implementation-agnostic specification. At its core, this specification describes the structure of an FDO and the rules that govern the relationships between its components. Section 5 defines the essential components and rules needed for creating and interpreting FDOs and the FDO type system. All components are themselves fully valid FDOs without any special mechanisms. This grounds FDOs in themselves to prevent unwanted dependencies from specific technologies and enable a common exchange format between implementations. In Section 6 we describe how a client finds, accesses, and interprets FDOs to get to the data represented by it. We expect that implementations of FDOs will vary and may not realize all of the building blocks. Thus, we specify in Section 7 how implementations adopt this document to facilitate interoperability between implementations.

An FDO consists of the following components (see [[[#fig1]]]):

  • Resource: Each FDO is astandardised and persistent bundle of information that represents (i.e., identifies, encapsulates, and describes) a resource. The resource could be digital, in which case the resource would usually be a data bit-sequence. But it is also possible that this resource is physical, in which case there is no data bit-sequence attached to the FDO. The resource is the heart of the FDO and all other components are there to make the resource and its metadata machine actionable.
  • Persistent identifier (PID): Each FDO is identified by a globally unique and persistent identifier. A PID system must resolve the PID to an FDO record.
  • FDO Record: The FDO record is a predictable and machine actionable information structure that contains attributes of the resource which the FDO represents. The FDO record is that piece of information a FDO client first encounters after PID resolution.
  • FDO attributes: The FDO record consists of FDO attributes in the form of tuples, where
    1. the first element of the tuple identifies an FDO Attribute Definition (which is similar to the definition of a metadata field in a metadata schema), and
    2. the second element is a value instantiating this FDO Attribute Definition (similar to the value that is filled into the metadata field).
    FDO attributes are properties of the resource the FDO represents, hence they are metadata. We distinguish three types of FDO attributes/attribute definitions:
    1. This document specifies mandatory FDO attributes that must be used to ensure predictability and machine actionability of any FDO.:
      • FDO type 0.FDO/Type
      • FDO profile (0. FDO/Profile)
      • FDO data (0. FDO/Data)
    2. The FDO Forum is maintaining a list of frequently used optional FDO attributes, that are formally defined and registered with PIDs. These can be used by FDO creators to increase interoperability between different implementations.
    3. Communities of practice are free to choose their own community-defined FDO attributes, however, they need to ensure that they are defined and registered in open registries.
  • FDO Attribute Definition: An FDO Attribute Definition specifies syntactic rules and a semantic meaning of an FDO attribute. In detail, an attribute definition specifies:
    1. a label for the attribute (e.g., “Author”),
    2. a semantic description of the attribute (e.g., “Person who wrote the document”),
    3. a cardinality (e.g., this attribute can be used at most one time in an FDO record), and
    4. syntactic rules for the value that instantiates this attribute (e.g., “a string that is at most 100 characters long”).
    Attribute definitions ensure that the attributes in the FDO record are machine interpretable. They also enable validation, i.e., to check that the values of attributes follow the syntactic rules specified in the attribute definitions. Each attribute definition is itself an FDO.
  • FDO Profile: The FDO profile specifies the set of FDO Attribute Definitions that must or can (depending on the cardinality of the attribute definitions) be instantiated to create FDO records by a certain repository, community, or implementation. The FDO profile makes the content of an FDO record predictable for an FDO client: Each FDO record needs to have a reference to the FDO profile it complies with. By inspecting the profile, a client learns which attributes it can expect to find in the FDO record. Each FDO profile is itself an FDO.
  • FDO Type: Each FDO has to specify an FDO type which essentially provides any client with a hint how the resource can be processed.
  • Registries: Attribute Definitions and profiles must be stored in open registries. This gives the possibility of a reverse lookup. The mandatory and optional attribute definitions are stored in registries maintained by the FDO Forum. Communities of practice maintain their own community-defined attribute definitions in open registries.
