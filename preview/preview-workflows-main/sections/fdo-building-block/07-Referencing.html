References are key to the FDO concept as they are used to associate bit-sequences and refer to any other related objects and information. However, their syntax, interpretation, and resolution remain implementation specific. This section aims at providing a solution for intra and inter FDO references.

References

[R7-1]
The most primitive syntax for all references MUST be a string.
[R7-2]
A reference is considered an unvalidated reference if it is a string that follows the syntactic format of a PID. The PID MAY not be resolvable. The classification as an “unvalidated reference” MAY be performed by a heuristic that creates false positives. Therefore, unvalidated references MAY be called a “PID candidate”, as they syntactically appear to be a PID, but MAY NOT always resolve to an FDO.
[R7-3]
For a reference to be considered a validated reference, it MUST resolve to a valid FDO record in reasonable time. It MUST also qualify as an “unvalidated reference”.

Profile Referencing

Profile referencing permits an attribute definition A to specify that the value aval (of an attribute a=(a def,aval) that instantiates A) MUST be a reference to an FDO following a specific profile. Hence, it is a mechanism to enforce that specific attributes in FDO records reference other FDOs (i.e., are validated references) according to a certain profile. Profile referencing works as follows:
The attribute definition 0.FDO/Profile uses profile referencing. It contains the following two attributes:
  • 0.FDO/ValidationMechanism, ProfileReference: This means that the attribute definition 0.FDO/Profile applies the profile referencing mechanism.
  • 0.FDO/DataType, 0.FDO/ProfileDef: This means that if the attribute definition 0.FDO/Profile is instantiated, the value must be a reference to an FDO that instantiates 0.FDO/ProfileDef (i.e., it must be a reference to a profile)

The Detailed Requirements for Profile Referencing are:

Format of an Attribute Definition that Applies Profile Referencing:
[R7-4]
An attribute definition MUST explicitly state that it uses this feature by containing the attribute (0.FDO/ValidationMechanism, ProfileReference).
[R7-5]
The attribute definition MAY also specify a list of profile PIDs P1,...,Pn where at least one of them MUST validate (see ) the referenced FDO. This is done by adding the attribute (0.FDO/DataType, Pi), i=1,...,n, into the attribute definition. If 0.FDO/DataType is not contained in the attribute definition, it just means that the referenced entity is an FDO (hence, that the value is a validated reference complying to at least 0.FDO/Root).
Instantiation of the Attribute Definition that Applies Profile Referencing:
[R7-6]
When the attribute definition is instantiated, the
  • first element MUST be either the PID or the name of the attribute definition, and
  • the value MUST be either
    • a validated reference to another FDO following at least one of the specified profiles, or
    • an in-line serialization of an FDO that instantiates at least one of the specified profiles.

Inline FDOs

When using profile referencing, an attribute references another FDO, for example a=(<attr_def>,<PID resolving to other FDO>). However, rather than referencing the other FDO, it may be serialized directly as the in-line value of a. In this case, the serialization depends on the implementation, and the implementation MUST specify the serialization format. Furthermore, inline-FDOs typically do NOT have their own PIDs and are hence NOT suited for attribute definitions, syntax definitions or profiles.

Attribute Referencing

Attribute referencing is a generic mechanism for redirecting to one or multiple other attribute(s) in the same record. Attribute referencing can, for example, be used to group information for different purposes. Or it can be used very similarly to unions, i.e., to state that an attribute is effectively realized through one or several other attributes in the record, which follow more specific attribute definitions. To start with, here is an example:
The mandatory attribute 0.FDO/Data (and actually also the 0.FDO/Type attribute) apply attribute referencing. The reason for this is that data can appear in different forms (see also [[[#attr-ref-1]]]):
  • Data may be given by a reference to an external bit-sequence. Then, the 0.FDO/Data attribute would have as its value a URL. ([[[#attr-ref-1]]], top-right)
  • Data could also be encapsulated in its own FDO, especially, if the FDO is a collection. Then, the 0.FDO/Data attribute would have as its value a reference to an FDO, which possibly follows a specific profile ([[[#attr-ref-1]]], bottom-left).
  • Data could also be in-line in the record. Then, the 0.FDO/Data attribute would have as its value a simple data bit-sequence ([[[#attr-ref-1]]], top-left).
As 0.FDO/Data uses attribute referencing, all these different notions of data references can be unified in one common and standardized attribute definition 0.FDO/Data. It is essential that the 0.FDO/Data attribute is not redefined over and over by different implementations. Otherwise, a machine would not be able to know which attribute to inspect in order to find the data bit-sequence of an FDO, irrespective of the implementation. Attribute referencing provides a harmonized entrypoint (0.FDO/Data) for machines to easily detect data.

The detailed requirements for Attribute Referencing are:

Format of an Attribute Definition that Applies Attribute Referencing:

The following requirements must be fulfilled such that an attribute definition A applies attribute referencing:

[R7-7]
The attribute definition A MUST explicitly state that it uses this feature by containing the attribute (0.FDO/ValidationMechanism, AttributeReference).
[R7-8]
The attribute definition A MAY specify a whitelist of attribute definition PIDs B1,...,Bn using the attribute (0.FDO/DataType, Bi), i=1,...,n, in its record. If A specifies this whitelist, an attribute a (instantiating A) MUST reference an attribute b that instantiates any of B1,...,Bn. If the 0.FDO/DataType attribute is not contained in A, any attribute b may be referenced by a, as there is no whitelist.
[R7-9]
The cardinality of the attribute definition A only specifies the occurrence of instances of A itself (and not the occurrence of instances of any of B1,...,Bn). The cardinality of B1,...,Bn cannot be influenced by A.

Instantiation of an Attribute Definition that Applies Attribute Referencing

[R7-10]
An attribute a=(adef,aval) (instantiating attribute definition A) is a reference to an attribute b=(bdef,bval) (instantiating attribute definition B) if and only if
  • aval is a PID resolving to an attribute definition
  • the referenced attribute b is instantiated in the same FDO record as attribute a
  • aval = bdef
  • bval is validated by bdef. This ensures, the reference is actually fulfilled.

See also [[[#attr-ref-2]]] and [[[#attr-ref-3]]] for illustration. If A is specified to apply attribute referencing (see [R7-7]) but there is no attribute b that fulfills all four conditions, then attribute a MAY be ignored to handle this error gracefully in certain use cases.

[R7-11]
There may be cases of consecutive referencing (e.g., a references b, b references c, …). All such references within an FDO record must form a directed acyclic graph (no loops allowed).
Note that attribute referencing is very similar to unions of attribute definitions (see [R5-9]): In both cases, an attribute definition states that its instantiation validates one of several attribute definitions. However, the slight difference is that for attribute referencing, the instantiating record also states which of the attribute definitions is actually chosen, while this is not stated in the case of unions.

External Vocabulary Referencing

It is essential for FDOs that existing vocabularies, ontologies and terminologies can be reused in records (even if they are not in the strict format as described in section 3.5). This can be carried out by applying external vocabulary referencing.
[[[#ex-vocab-ref]]] shows how schema.org terms can be used in an FDO record by applying external vocabulary referencing.
  • The attribute definition 0.FDO/SchemaOrgTerm states that it uses external attribute referencing through the attribute (0.FDO/ValidationMechanism, ExternalVocabularyReference).
  • The syntax definition 0.FDO/SchemaOrgTermSyntax specifies the syntax of the attribute definition, i.e., it must be instantiated by the string “https://schema.org”, which is the schema.org namespace.
  • The FDO 21.123/456 contains the attribute (0.FDO/SchemaOrgTerm, https://schema.org) to indicate that it uses vocabulary from schema.org. Afterwards, any schema.org term can be used as an attribute in the form (<URL of term>, <value>), such as (https://schema.org/priceCurrency, USD).

The detailed requirements for External Vocabulary Referencing are:

Format of an Attribute Definition that Applies External Vocabulary Referencing:

The following detailed requirements must be fulfilled such that an attribute definition A applies external vocabulary referencing:

[R7-12]
The attribute definition A MUST explicitly state that it uses this feature by containing the attribute (0.FDO/ValidationMechanism, ExternalVocabularyReference).
[R7-13]
The attribute definition A MUST specify the namespace of the external vocabulary using the attribute 0.FDO/DataType which references a syntax definition that validates the namespace.

Instantiation of External Vocabulary Attributes in the Record:

[R7-14]
To use external vocabulary in the FDO record, the record MUST contain
  • an attribute a=(adef,aval) that instantiates attribute definition A, where A applies external vocabulary referencing
  • an attribute b=(bdef,bval) representing a vocabulary term
The following requirements must be fulfilled by a and b:
  • aval is a URL representing a namespace
  • bdef = aval/<term>, i.e., bdef is a URL representing a term under a namespace
  • bval instantiates bdef.
