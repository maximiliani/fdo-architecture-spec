This step determines whether the given FDO record’s type is that which the client expects while processing the FDO. It assumes that [Step 4] was completed and that its status indicates that a 0.FDO/Type field is present.

There are many reasons to perform this step. It could be that the client software is only capable of processing limited types of information and the quicker it can determine whether the FDO is appropriate the better.

Also, the accessing client may determine when to check for an FDO’s type as it could be using different hints to determine whether or not the FDO is appropriate.

Inputs
This step expects that [Step 4] had a FDO Root Profile validation status of COMPLETE, REQUIRED_FIELDS, TYPE_ONLY and two inputs:
  • The list of types that the client is looking for. This is called the expected FDO Type list. It can be empty indicating that any FDO Type within the FDO Record will be accepted. This forces the MATCH_ALL flag described below.
  • The Type_Matching flag. There are three possible values for this flag:
    • MATCH_ALL. Indicates that any FDO type is acceptable. This ignores the Expected FDO Type list if provided.
    • EXACT_MATCH. Indicates that the FDO Type(s) in the FDO record need to match exactly the FDO types in the expected FDO Type list.
    • DERIVED_MATCH. Indicates that the FDO Type can be derived from the FDO Type found in the expected FDO Type list. Derived types are backward compatible with their parent types.
Outputs
This step returns the status of the match as well as a list of the FDO Types that match or are derived from the expected types. If the list is empty, it indicates that none were found. The client can determine from the number of FDO Type matches whether the match is full or partial.
Expected behavior
In this step, the client will compare the list of FDO Types in the FDO Record with the expected FDO Type list looking for matches. The following lists the possible scenarios:
  1. If the FDO Type List is empty and/or the Type_Matching flag is set to MATCH_ALL, all the FDO Types within the FDO Record are considered appropriate. The result is FULL_MATCH and the list of all FDO Types found in the FDO Record.
  2. The FDO Types in the FDO records fully match those in the expected FDO Type List. The result is ALL_MATCH and the list of all matched FDO Types.
  3. The FDO Types in the FDO record matches at least one of the FDO Type in the expected FDO Type list. The result is PARTIAL_MATCH and the list of the matching FDO Types.
  4. The FDO Type in the FDO records match none of the FDO Type list. There are two different results depending on the the matching condition
    • If the Type_Matching is set to EXACT_MATCH, the client concludes that the FDO does not have the proper type. The results is NO_MATCH.
    • If the Type_Matching is set to DERIVED_MATCH, the client will respectively analyse each of the FDO Type in the FDO Record and determine if they are derived from types that are within the expected FDO Type list. The assumption is that derived FDO Types are compatible with their parent FRO Type. To do so, the client resolves each of the FDO Types into their respective FDOs and then analyses these FDOs to see if the types they describe are derived from FDO Types that match a type within the FDO Type List. If the client determines that one or more of the FDO Type in the FDO Records are derived from FDO Types that are in the expected FDO Type list, it will conclude that the FDO Record is compatible with one or more of its expected FDO types. The results can be NO_MATCH if no derived FDO Types are found or DERIVED_MATCH if one or more derived FDO Types matches are found.
