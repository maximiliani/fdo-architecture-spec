The Generic FDO Processing Block consists of the discrete set of steps that a canonical FDO client would execute in order to access an FDO and all its components given its PID.

To help visualize the function of the FDO Processing block, it is useful to consider the simplest canonical FDO case illustrated in [[[#step-fig1]]] below as well as a more complex example in [[[#step-fig2]]]. It is important to note that both these figures are abstract representations of FDOs and different implementations could have these various attributes named, structures, and stored differently.

Abstract FDO Processing Block steps
All FDO Processing block implementations regardless of their specific data and implementation configurations SHOULD be able to describe how their approach satisfies the following set of discreet steps listed below.
[Step 1] PID Detection
Given a string, determine if it is a PID that can be resolved.
[Step 2] PID Processing / Resolution
Resolve the identifier into a PID record.
[Step 3] PID Record Parsing into an FDO Record
Parse the PID record in a set of type value pairs
[Step 4] FDO Record root profile validation
Locate all required core FDO attributes and create an FDO Record.
[Step 5] FDO Record Type inspection
Determine if the FDO has a type and whether this type is appropriate given the requirements of the client. A useful hint.
[Step 6] FDO Profile Inspection
Determine if the FDO has one or more profiles and whether these profiles are appropriate given the requirements of the client.
[Step 7] FDO Profile(s) Validation
Validate the FDO according to the profile(s) that are of interest to the client.
[Step 8] FDO Attributes Processing
The client validates each of the attributes values of interests and processes these values to confirm the appropriateness of the FDO.
[Step 9] FDO Attributes Evaluation
Evaluates the values of the attributes to make sure that they are appropriate.
[Step 10] FDO Data Access
The client accesses the FDO’s data subject to access control.
[Step 11] FDO Record Type Validation
Determine if the FDO’s data matches the type of the FDO.
This specification uses these steps to describe the function and characteristic of the various components of the FDO. FDO implementations are not required to implement each of these steps as long as they can account for how their implicit assumptions and different components achieve the same results.
