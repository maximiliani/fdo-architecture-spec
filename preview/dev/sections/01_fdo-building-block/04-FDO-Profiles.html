FDO Profiles

FDO Profiles are templates for FDO Records. They specify which attributes SHOULD be considered in an FDO Record. They DO NOT specify the cardinality of the attributes; this is done in the attribute definitions. All FDO Profiles are themselves fully valid FDOs (see [R4-6]) that instantiate the profile definition profile (see [R4-7]). All FDO Profiles MUST fulfill the following requirements:

[R4-1] Definition of the term "FDO Profile"

An FDO Profile (short: profile) is a machine-readable template that MAY be instantiated by FDO Records. It defines a set of attribute definitions that MUST be contained in any FDO Record instantiating the profile, according to the cardinalities specified in the attribute definitions (see [R5-7]).

[R4-1] Attribute Definition Inclusion

An attribute definition is contained in a profile’s list of attribute definitions (as described in [R4-1]) if and only if the profile contains an attribute, where the attribute’s
  • first element is either 0.FDO/Attribute or 0.FDO/Attribute, and
  • the second element (value) contains a PID resolving to the attribute definition.

[R4-3] Mandatory Profile attributes

Each profile MUST at least contain the mandatory attributes 0.FDO/Type, 0.FDO/Profile, and 0.FDO/Data as defined by the FDO Forum (see the root profile). It can further contain attributes from the list of optional attributes as defined by FDO Forum (see table X in appendix), or arbitrary community-specified attributes, as long as these are identified by PIDs.

[R4-4] Additional attributes

An FDO Record MAY contain additional attributes whose attribute definitions are not listed in the profiles it instantiates. Profiles MAY prohibit this feature for FDOs that instantiate them by setting the value of the optional 0.FDO/denyAdditionalAttributes attribute to a truthy value. By default, additional attributes are allowed to facilitate the use of multiple FDO profiles in a single FDO Record. Explicitly prohibiting additional attributes SHOULD be sparingly used and is NOT recommended.

[R4-5] Profile Compliance

Each FDO Record MUST comply with all the profiles it instantiates. This means that it MUST successfully validate (see ) against these profiles at least upon creation and update of the FDO record.

[R4-6] Profiles are FDOs

Each profile is itself represented as an FDO. Hence, it is identified by a PID that resolves into an FDO Record consisting of attribute tuples. All profiles MUST at least comply with the FDO root profile and the FDO Profile definition profile (see [R4-7]).

[R4-7] Profile Definition Profile

The FDO Profile definition profile defines a set of attribute definitions that MUST be instantiated in the FDO Record of each profile. Thus, this profile for all profile FDOs, is itself a fully valid FDO that instantiates the root profile and itself. It uses profile extension to inherit the set of attribute definitions of the root profile. In addition to those attributes, the FDO Profile definition profile also demands the use of 0.FDO/Name and 0.FDO/Description. The 0.FDO/Attribute attribute is used to specify the set of attribute definitions that MUST be used in any FDO instantiating a profile (profile FDOs included). 0.FDO/ExtendsProfile is used by the profile extension mechanism to indicate the "parent" profile to extend from. The conceptual record of the FDO Profile definition profile with the PID 0.FDO/ProfileDef is defined in Table 1.

[R4-8] Profile Registration

Any profile and all attribute definitions it uses MUST be defined in open registries. This is done to promote the findability and reusability of FDO profiles, and thus the interoperability of FDOs. Special rules apply in case profiles are updated, see R11-3 - R11-6.

The FDO Root Profile (0.FDO/Root) can be extended to create new FDO Profiles:

Since profiles are defined as lists of attribute definitions, there exists a simple mechanism to extend profiles. The mechanism is defined as follows:

[R4-9] Root Profile

The root profile defines a minimal set of attribute definitions that MUST be instantiated in any FDO Record. All FDOs including the root profile itself MUST comply with the root profile. It is the minimal profile in the sense that it contains the mandatory attribute definitions 0.FDO/Type, 0.FDO/Profile, and 0.FDO/Data that are essential to the interpretability of FDO records. Therefore, compliance with the root profile MUST be checked by all clients even if it is not explicitly included in an FDO Record or its profile extension hierarchy. The root profile is defined by the FDO Forum and has the PID 0.FDO/Root. It instatiates itself and 0.FDO/ProfileDef, as it is a profile, and contains the list of attribute definitions shown in Table 2.

[R4-10] Profile Extension Mechanism

Profiles SHOULD be extended by other profiles whenever semantically appropriate. The root profile serves as the root of the extension mechanism, meaning, any profile MUST be defined as an extension of the root profile or any of its children.
To extend a profile P1 from a profile P2, the P2 MUST contain an attributewhere
  • the first element is 0.FDO/ExtendsProfile or the string FDO_Extends_Profile, and
  • the value is the PID identifying P1.
In this case, P2 inherits all attribute definitions listed as values to 0.FDO/Attribute in P1. This is recursive, such that the attribute definitions that are listed in profiles that P1 extends from also apply to P2. P2 MAY add further attribute definitions by instantiating 0.FDO/Attribute in its FDO Record. The resulting inheritance hierarchy MUST be conflict-free and representable by a directed acyclic graph.

[R4-11] Profile Validation Hierarchy

To validate an FDO Record against a profile that uses the extension mechanism, all profiles in the extension hierarchy MUST be resolved until a profile is reached that does not extend another profile (which may be 0.FDO/Root or another profile). During this process, a set of attribute definitions is collected from all profiles in the extension hierarchy. The FDO Record MUST then be validated against this set of attribute definitions according to the cardinalities specified in the attribute definitions. If any of the profiles in the extension hierarchy explicitly prohibits additional attributes, then the FDO Record MUST NOT contain any attributes that are not listed in the set of attribute definitions up to this node in the hierarchy. The validation only succeeds if all attribute definitions are satisfied and none of the profiles in the extension hierarchy prohibits additional attributes that are present in the FDO Record.
