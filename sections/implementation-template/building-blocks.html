FDO Profiles

[I4-1] Profile representation
Each profile is itself an FDO. You must specify how the FDO profile is represented in the given implementation (i.e., how the FDO looks like; this can be understood as a recipe for building FDO profiles in the given FDO implementation).
Note: In the validations chapter, you need to define an algorithm for a machine to read and interpret a profile in the given implementation in order to perform validation of records against the profile.

FDO Attribute Definitions

[I5-1] Representation of FDO attribute definitions (M)
Each attribute definition is itself an FDO. You must specify how the attribute definition is represented in the given implementation (i.e., how the attribute definition looks like; this can be understood as a recipe for building FDO attribute definitions in the given FDO implementation).
Note: [I5-1] should define the overall structure of an attribute definition. You can go into more detail of how to encode syntax, semantics, complex attributes and complex validation rules in [I5-3] - [I5-8]. If your implementation uses additional implementation-specific rules for attributes or validation, you can describe them in [I5-9] - ...
[I5-2] Attribute Definition Profile (M)
Give the PID for the attribute definition profile that is used in your implementation.
[I5-3] Encoding Semantics (M)
Specify where one can find the name and the description of an attribute definition.
[I5-4] Encoding Syntax (M)
Specify how the following rules are represented in an attribute definition in the given implementation. An attribute…
  • … must have a cardinality.
  • … must follows one of the primitive data types “string”, “number”, “integer” and “boolean”.
  • … may comply with a regular expression.
  • … may lie within a numeric range.
  • … may be equal to one element from a whitelist.
  • … may be not equal to all elements from a blacklist.
Note: If some of these rules are not compatible with the given implementation (i.e., cannot be encoded in the given technology), you must point this out. Different technologies may have different levels of expressability.
[I5-5] Composition of Attribute Definitions (R)
Specify how compositions of attributes are represented in the given implementation.
[I5-6] Unions of Attribute Definitions (R)
Specify how unions of attributes are represented in the given implementation.
[I5-7] Profile Referencing (R)
Specify how profile referencing looks like in the given implementation.
[I5-8] Attribute Referencing (R)
Specify how attribute referencing looks like in the given implementation.
[I5-9] - ... Implementation-specific rules (O)
You can further specify semantic, syntactic, and complex validation rules that are not covered by this document. Then, you must describe how the rules are encoded in the given implementation, and how a record is validated against the rules.
