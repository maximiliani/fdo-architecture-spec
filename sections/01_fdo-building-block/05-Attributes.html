FDO Attributes

FDO Attributes are properties of an FDO:
[R5-1] Attribute Definition Tuple
Each tuple in an FDO Record is called an FDO Attribute (short: attribute).
[R5-2] Attribute Structure
An attribute denotes a property of an FDO in form of a tuple that consists of the following two elements:
  • The first element of the tuple MUST be either a PID that resolves to an attribute definition, or a human-readable name of an attribute.
  • The second element (called value) can be any literal. If the first element refers to an attribute definition, then the value MUST be successfully validated (see ) against the attribute definition at least upon creation and update.
[R5-3] Attribute Categories
There are three qualitative categories of attributes (or attribute definitions): mandatory, optional, and community-defined attributes.
  • The FDO Forum defines the mandatory attributes. These attributes MUST be used in each FDO Record. This set is defined in the root profile (see ).
  • The FDO Forum defines the optional attributes. It is a list of pre-defined attributes which are frequently reused by use-cases within their FDO Records and profiles (see Table X in appendix). Use-cases SHOULD reuse these attributes, whenever suitable, for better harmonization of FDO information.
  • Community-defined attributes are any attributes that are not defined by the FDO Forum. They appear in different forms, with decreasing level of machine-actionability in the following enumeration:
    • A Community attribute MAY be specified as an attribute definition, according to [R5-4] - [R5-15], against which the instantiated attribute in an FDO Record MUST be validated. The attribute definition is instantiated as the following attribute:
      • first element: the PID or the name of the attribute definition
      • value: any literal validating against the first element
    • Community attributes MAY be used in records by referencing existing elements in vocabularies, terminologies, or ontologies. Validation MAY be performed, but is not guaranteed. The attribute looks as follows:
      • first element: reference to element from vocabulary, terminology, or ontology
      • value: instantiates the first element
      See for more details.
      • Community attributes MAY be used in records by a simple human-readable label. No validation can be performed here.
      • first element: human-readable label
      • value: any literal

    These levels do not impose additional requirements but help repositories and users to assess validation behavior, interoperability, and long-term interpretability of FDO Records.

An FDO attribute definition is a template for FDO Attributes:
[R5-4] Attribute Definitions are FDOs
FDO attribute definitions (short: attribute definitions) are themselves represented as FDOs. Hence, they are identified by a PID that resolves into an FDO Record consisting of attributes.
[R5-5] Syntactic Rules
An attribute definition specifies syntactic, machine-readable rules for an attribute in an FDO Record (that is, the attribute instantiates the attribute definition).
[R5-6] Semantic Description
Attribute definitions describe the semantics of an attribute. They MUST provide at least one localized and human-readable name. For better understanding, at least one localized and human-readable description SHOULD be provided.
[R5-7] Cardinality Specification
An attribute definition MUST specify the cardinality of an attribute. The cardinality MUST follow this regular expression: ^(\d+)(\.\.(\d+|\*))?$. Examples for possible cardinalities include but are not limited to:
  • 1 (mandatory),
  • 0..1 (optional),
  • 1..* (mandatory and repeatable),
  • 0..* (optional and repeatable),
  • 2..3 (MUST occur two to three times).
[R5-8] Syntax Reference
To specify syntactic, machine-readable rules, an attribute definition MUST contain a syntax definition by using the attribute 0.FDO/DataType with a reference to a syntax definition (see ) as its value. Furthermore, the attribute definition MUST contain the attribute (0.FDO/ValidationMechanism, Syntax). To instantiate the attribute definition in an FDO Record,
  • the first element MUST be either the PID or a name of the attribute definition, and
  • the value MUST validate against the syntax definition referenced in the attribute definition at least upon creation and update (see ).
[R5-9] Union Validation
An attribute definition A MAY be a union of attribute definitions (a.k.a. “any of”) A1,...,An by containing the attribute (0.FDO/ValidationMechanism, Union) and the attributes (0.FDO/DataType, Ai) for i=1,...,n. To instantiate this attribute definition in another FDO record,
  • the first element MUST be the PID or a name of A,
  • the value MUST comply with the syntactic rules of at least one of A1,...,An.
[R5-10] Composition Validation
An attribute definition A MAY be a composition of attribute definitions (a.k.a. “all of”) A1,...,An by containing the attribute (0.FDO/ValidationMechanism, InlineCombination) and the attributes (0.FDO/DataType, Ai) for i=1,...,n. To instantiate this attribute definition in another FDO record,
  • the first element MUST be the PID or a name of A,
  • the value MUST be an inline combination of instantiations of A1,...,An. The format of this inline combination (e.g. JSON, JSON-LD, …) MAY depend on the implementation.
[R5-11] Advanced Referencing Mechanisms
There exist three pre-defined, advanced mechanisms for attribute definitions which follow special patterns:
  • Profile referencing
  • Attribute referencing
  • ExternalVocabularyReferencing
These mechanisms are further specified in and MAY be applied by attribute definitions instead of [R5-9] - [R5-11].
The FDO attribute definition profile (0.FDO/AttributeDef) is the template for all FDO attribute definitions:
[R5-12] Attribute Definition Profile
The FDO attribute definition profile defines a list of attribute definitions that MUST be instantiated in the FDO record of each attribute definition, according to the cardinalities in the attribute definitions (i.e., each attribute definition instantiates the FDO attribute definition profile). The FDO attribute definition profile is defined by the FDO Forum, has the PID 0.FDO/AttributeDef and contains the list of attribute definitions shown in Table 3.
[R5-13] Additional Attribute Definition Attributes
Attribute definitions MAY have additional attributes and instantiate additional profiles for any reason.
[R5-14] Forum Attribute Definitions
The attribute definitions listed in Tables 1 - 4 (comprising the mandatory and optional attributes defined by the FDO Forum) are implemented within the Data Type Registry (DTR) by the FDO Forum. Those attribute definitions are implemented as Handles and SHOULD be reused in any FDO implementation independent of the used technology. This ensures interoperability between different FDO implementations and compliance with the type system.
[R5-15] Community Attribute Definitions
Community-specified attribute definitions MAY be implemented by communities of practice in the DTR or any other suitable open registry. Special rules apply when attribute definitions should be updated, see [R11-3] - [R11-6].
