FDO Types

Typing is an essential building block of the FDO concept. The term “type/typing” refers to several concepts simultaneously:

  1. Attribute definitions are types for properties of an FDO and are instantiated by attributes in the FDO record.
  2. The value of the 0.FDO/Type attribute in the FDO record is a type of the resource being represented by the FDO.
  3. The value of the 0.FDO/Profile attribute in the FDO record is a type of the FDO record (which instantiates the profile).
All these types together form the type system.

The FDO Type

This section specifies the (declared) FDO Type (0.FDO/Type) as the value of the 0.FDO/Type attribute, which is instantiated in FDO records (2. In above enumeration). This 0.FDO/Type attribute is an intentionally chosen entry point that signals some default processing semantics for the FDO. Other facets of the FDO type could be derived independently of the 0.FDO/Type attribute using other typing mechanisms, e.g., by inspecting the presence of other record attributes . Please note that we will refer to the 0.FDO/Type attribute as the “FDO type” in the following.

Purpose of the FDO Type:
[R10-1] Type Attribute Requirement
0.FDO/Type is a mandatory attribute that MUST be instantiated in each FDO record. It can be repeated multiple times. It specifies the type of the resource being represented by the FDO.
[R10-2] Type Processing Hint
The purpose of the 0.FDO/Type attribute is to give a hint to a machine how this object MAY be processed.
The FDO Type of FDOs that belong to the Type System:
[R10-3] Type System FDO Types
Profiles, attribute definitions, syntax definitions, and FDOs representing the FDO type (see [R10-6]) are special FDOs in the sense that they are part of the type system. They MUST have the following FDO type:
  • A profile has the FDO type “FDO_Profile”.
  • An attribute definition has the FDO type “FDO_Attribute_Definition”.
  • A syntax definition has the FDO type “FDO_Syntax_Definition”.
  • An FDO representing an FDO type has the FDO type “FDO_Type_Definition”.
Adding additional FDO types to these FDOs is allowed.
FDO Types are defined by communities:
[R10-4] Community Type Definition
Communities of practice MAY define their FDO type(s) according to their needs.
[R10-5] Type Interpretability
The FDO type SHOULD also be interpretable outside the community of practice to ensure interoperability. This SHOULD be ensured by linking to existing standards or software that explain how to interpret the FDO type. These external resources SHOULD be publicly accessible.
[R10-6] Type as FDO
Community-defined FDO type(s) MAY be FDOs for better interpretability.
[R10-7] Type Derivation
Types MAY be derived from other types (i.e., there may be hierarchies of types defined within communities of practice). To state that a type t2 is derived from a type t1, the FDO record of t2 SHOULD contain the attribute 0.FDO/DerivedFromType, whose value is a PID that resolves to t1.
FDO Type of collections:

FDOs may represent collections of objects (see also ). To assign FDO type(s) to a collection and to objects within the collection, the following conditions apply:

[R10-8] Collection Type
The FDO type(s) of a collection MUST describe the collection as a whole.
[R10-9] Object Type in Collection
If an FDO type SHOULD be assigned to objects within the collection, each object MUST be represented as an FDO itself. This FDO can then contain the 0.FDO/Type attribute to specify FDO type(s).

Operation Association Mechanisms

Types MAY be used to associate FDOs to operations. The specification of FDO operations is still in an early state and not yet completely understood. A more comprehensive list of requirements beyond [R10-10] – [R10-13] will be included in a later version of this document. However, the specification of how to associate FDO operations with FDOs which they are able to act on is already more elaborate. The following requirements apply:

FDO Operations:
[R10-10] Operation Definition
FDO operations (short: operations) are procedures that can be invoked on FDOs to create, manipulate or delete FDO record(s), bit-sequence(s), or trigger other related activities.
[R10-11] Operation Interface
Operations abstract and encapsulate their concrete implementation details by providing a uniform interface that at least defines their input parameters, output parameters, and how the operation should be executed. This enables execution of operations in arbitrary, possibly restricted environments.
[R10-12] Operation as FDO
Operations MUST themselves be represented as FDOs. Hence, they are identified by a PID that resolves into an FDO record consisting of attributes.
[R10-13] Operation Definition Profile
An operation definition profile 0.FDO/OperationDef which specifies a minimal set of attributes that an operation must define (similar to 0.FDO/ProfileDef or 0.FDO/AttributeDef, see and ) does not exist yet but may be specified in the future.
FDO Operation Association Mechanisms:
[R10-14] Operation Association
To associate an operation to an FDO, types SHOULD be used (i.e., FDO attribute(s), FDO profile(s) and the FDO type(s) as defined in ). In detail, to associate an operation to an FDO, (at least) one of the following operation association mechanisms SHOULD be applied:
  • The FDO record refers to applicable operations for this FDO via a designated attribute.
  • The FDO profile refers to applicable operations for any FDO complying to this profile via a designated attribute.
  • The attributes available in an FDO record are matched with applicable operations and the attributes they require.
More information on the operation association mechanisms are available in the paper “A Comparative Analysis of Modelling Approaches for the Association of FAIR Digital Object Operations” 10.5334/dsj-2025-022.
[R10-15] List Operations
Providing an operation to list available operations for specific FDO types is strongly RECOMMENDED.
