Advanced Referencing

[R7-1] References

References are key to FDOs as they are used to associate bit-sequences and refer to any other related objects and information. Therefore, the primitive data type for all references MUST be a string. However, their syntax, interpretation, and resolution remain implementation specific. This section aims at providing a solution for intra and inter FDO references.

[R7-2] Unvalidated References

A reference is considered an unvalidated reference if it is a string that follows the syntactic format of a PID. The PID MAY not be resolvable. The classification as an unvalidated reference MAY be performed by a heuristic that creates false positives. Therefore, unvalidated references MAY be called a PID candidate, as they syntactically appear to be a PID, but MAY NOT always resolve to an FDO.

[R7-3] Validated References

For a reference to be considered a validated reference, it MUST resolve to a valid FDOrecord in reasonable time. It MUST also qualify as an unvalidated reference.

Profile Referencing

Profile referencing permits an attribute definition A to specify that the value aval (of an attribute a=(adef,aval) that instantiates A) MUST be a reference to an FDO following a specific profile. Hence, it is a mechanism to enforce that specific attributes in FDO records reference other FDOs (i.e., are validated references) that comply with a certain profile. Profile referencing works as follows:
The attribute definition 0.FDO/Profile uses profile referencing. It contains the following two attributes:
  • (0.FDO/ValidationMechanism, ProfileReference: This means that the attribute definition with the PID 0.FDO/Profile applies the profile referencing mechanism.
  • (0.FDO/DataType, 0.FDO/ProfileDef): This means that if the attribute definition 0.FDO/Profile is instantiated, the value must be a reference to an FDO that instantiates 0.FDO/ProfileDef (i.e., it must be a reference to an FDO profile)

The Detailed Requirements for Profile Referencing are:

Format of an Attribute Definition that Applies Profile Referencing:

[R7-4] Declaring a profile reference

An attribute definition MUST explicitly state that it uses this feature by containing the attribute (0.FDO/ValidationMechanism, ProfileReference).

[R7-5] Profile Reference Target

The attribute definition MAY also specify a list of profile PIDs P1,...,Pn where at least one of them MUST validate the referenced FDO. This is done by adding the attribute (0.FDO/Data, Pi), i=1,...,n, into the attribute definition. If 0.FDO/Data is not contained in the attribute definition, it just means that the referenced entity is an FDO (hence, that the value is a validated reference complying to at least 0.FDO/Root).
Instantiation of the Attribute Definition that Applies Profile Referencing:

[R7-6] Profile Reference Instantiation

When the attribute definition is instantiated (i.e., it is used in an FDO with at least one value), the
  • first element MUST be either the PID or the name of the attribute definition, and
  • the value MUST be either
    • a validated reference to another FDO following at least one of the specified profiles, or
    • an inline serialization of an FDO that instantiates at least one of the specified profiles.

Inline FDOs

When using profile referencing, an attribute references anotherFDO, for example a=(<attr_def>,<PID resolving to other FDO>). However, rather than referencing the other FDO, it may be serialized directly as the in-line value of a. In this case, the serialization depends on the implementation, and the implementation MUST specify the serialization format. Furthermore, inline-FDOs typically do NOT have their own PIDs and are hence NOT suited for attribute definitions, syntax definitions or profile definitions.

Attribute Referencing

Attribute referencing is a generic mechanism for redirecting to one or multiple other attribute(s) in the same record. Attribute referencing can, for example, be used to group information for different purposes. Or it can be used very similarly to unions, i.e., to state that an attribute is effectively realized through one or several other attributes in the record, which follow more specific attribute definitions. To start with, here is an example:

The mandatory attribute 0.FDO/ProfileDef (and actually also the 0.FDO/ProfileDef attribute) apply attribute referencing. The reason for this is that data can appear in different forms but need to be automatically recognized (see also ):
  • Data may be given by a reference to an external bit-sequence. Then, the 0.FDO/Data attribute would have as its value a URL. (, top-right)
  • Data could also be encapsulated in its own FDO, especially if the FDO is a collection. Then, the0.FDO/Data attribute would have as its value a reference to an FDO, which possibly follows a specific profile (, bottom-left).
  • Data could also be in-line in the record. Then, the 0.FDO/Data attribute would have as its value a simple data bit-sequence (, top-left).
As 0.FDO/ProfileDef uses attribute referencing, all these different notions of data references can be unified in one common and harmonized attribute definition 0.FDO/ProfileDef. It is essential that the 0.FDO/ProfileDef attribute is not redefined over and over by different implementations. Otherwise, a machine would not be able to know which attribute to inspect to find the data bit-sequence of an FDO, irrespective of the implementation. Attribute referencing provides a harmonized entrypoint (0.FDO/ProfileDef) for machines to easily detect data.

The detailed requirements for attribute referencing are:

Format of an Attribute Definition that applies attribute referencing:

The following requirements must be fulfilled such that an attribute definition A applies attribute referencing:

[R7-7] Attribute Reference Declaration

The attribute definition A MUST explicitly state that it uses this feature by containing the attribute tuple (0.FDO/ValidationMechanism, AttributeReference).

[R7-8] Attribute Reference Whitelist

The attribute definition A MAY specify a whitelist of attribute definitionPIDs B1,...,Bnusing the attribute (0.FDO/DataType, Bi), i=1,...,n, in its record. If A specifies this whitelist, an attribute a (instantiating A) MUST reference an attribute b that instantiates any of B1,...,Bn. If the 0.FDO/DataType attribute is not contained in A, a may reference any attribute b, as there is no whitelist.

[R7-9] Attribute Reference Cardinality

The cardinality of the attribute definition A only specifies the occurrence of instances of A itself (and not the occurrence of instances of any of B1,...,Bn). The cardinality of B1,...,Bn cannot be influenced by A.

Instantiation of an attribute definition that applies attribute referencing

[R7-10] Attribute Reference Instantiation

An attribute a=(adef,aval) (instantiating attribute definition A) is a reference to an attribute b=(bdef,bval) (instantiating attribute definition B) if and only if
  • aval is a PID resolving to an attribute definition
  • the referenced attribute b is instantiated in the same FDO record as attribute a
  • aval = bdef
  • bval is validated by bdef. This ensures the reference is actually fulfilled.

See also and for illustration. If A is specified to apply attribute referencing but there is no attribute b that fulfills all four conditions, then attribute a MAY be ignored to handle this error gracefully in certain use cases.

[R7-11] Consecutive usage of attribute references

There may be cases of consecutive referencing (e.g., a references b, b references c, …). All such references within an FDO record must form a directed acyclic graph (no loops allowed).

Note that attribute referencing is very similar to unions of attribute definitions: In both cases, an attribute definition states that its instantiation validates one of several attribute definitions. However, the slight difference is that for attribute referencing, the instantiating record also states which of the attribute definitions is actually chosen, while this is not stated in the case of unions (see ). If compared to programming principles, attribute references behave somewhat like pointers, whereas unions behave like union types.

  • The FDO 21.123/456 contains the attribute (0.FDO/SchemaOrgTerm, https://schema.org) to indicate that it uses vocabulary from schema.org. Afterward, any schema.org term can be used as an attribute in the form (<URL of term>, <value>), such as (https://schema.org/priceCurrency, USD).

The detailed requirements for external vocabulary referencing are:

Format of an Attribute Definition that applies external vocabulary referencing:

The following detailed requirements must be fulfilled such that an attribute definition A applies external vocabulary referencing:

[R7-12] External Vocabulary Declaration

The attribute definition A MUST explicitly state that it uses this feature by containing the attribute (0.FDO/ValidationMechanism, ExternalVocabularyReference).

[R7-13] Vocabulary Namespace

The attribute definition A using external vocabulary referencing MUST explicitly specify the namespace of the external vocabulary using the attribute 0.FDO/DataType which references a syntax definition that validates the namespace.

[R7-14] Using external vocabulary attributes in an FDO

To use external vocabulary referencing in the FDO record, the record MUST contain
  • an attribute a=(adef,aval) that instantiates attribute definition A, where A applies external vocabulary referencing
  • an attribute b=(bdef,bval) representing a vocabulary term
The following requirements must be fulfilled by a and b:
  • aval is a URL representing a namespace
  • bdef = aval/<term>, i.e., bdef is a URL representing a term under a namespace
  • bval instantiates bdef.
