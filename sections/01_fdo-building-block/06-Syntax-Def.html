Syntax Definitions

An FDO syntax definition specifies syntactic validation rules for FDO attributes:
[R6-1] Syntax Definition Purpose
FDO syntax definitions (short: syntax definitions) define syntactic validation rules for attribute definitions. They are referenced by attribute definitions (see [R5-8]).
[R6-2]
The following rules can be used to define a syntax: An attribute value can be specified to
  • follow one of the primitive data types “string”, “number”, “integer” and “boolean”.
  • comply with a regular expression.
  • lie within a numeric range.
  • be equal to one element from a whitelist.
  • be not equal to all elements from a blacklist.
[R6-4] shows how to encode these rules.
The FDO syntax definition profile (0.FDO/SyntaxDef) is the template for all FDO syntax definitions:
[R6-3] Syntax Definition as FDO
Syntax definitions are themselves represented as FDOs. Hence, they are identified by a PID that resolves into an FDO record consisting of attributes.
[R6-4]
The FDO syntax definition profile defines a list of attribute definitions that MUST be instantiated in the FDO record of each syntax definition, according to the cardinalities in the attribute definitions (i.e., each syntax definition instantiates the FDO syntax definition profile). The FDO syntax definition profile is defined by the FDO Forum, has the PID 0.FDO/SyntaxDef and contains the list of attribute definitions shown in Table 4.
[R6-5] Domain-Specific Validation
Syntax definitions MAY have additional attributes and instantiate additional profiles for any reason (e.g., for communities to add other syntactic rules than given in [R6-2]). Domain-specific validation technologies MAY be expressed using this; however their realization MUST be considered optional for generic clients.
[R6-6] Community Syntax Definitions
Community-specified syntax definitions MAY be implemented by communities of practice in the DTR or any other suitable open registry. Special rules apply when syntax definitions should be updated, see [R11-3] - [R11-6].
Table 4: The list of attribute definitions specified by the FDO syntax definition profile (0.FDO/SyntaxDef). These attributes MUST be used when defining any new syntax definition. This table does not include all information about the attributes. Please resolve the attribute definition PIDs via the Handle System or see the appendix of this document.
PID of Attribute Definition Description of the attribute Cardinality of the attribute
0.FDO/Type The FDO type(s) as specified in section 3.10. The FDO type of a syntax definition is simply the string FDO_Syntax_Definition. 1..*
0.FDO/Profile The profile(s) as specified in section 3.4. The profile of a syntax definition is 0.FDO/SyntaxDef. 1..*
0.FDO/Data Data of the FDO. For syntax definitions, this attribute does often not apply, in which case this attribute MAY simply use the string Not_Applicable. 1..*
0.FDO/Name Human-readable and localized labels for this syntax. 1..*
0.FDO/Description Human-readable and localized descriptions for this syntax. 0..*
0.FDO/PrimitiveDataType There are four possible values: “string”, “number”, “integer”, and “boolean”. All of these primitives MUST be supported by all clients. 1
0.FDO/Regex Only applicable if 0.FDO/PrimitiveDataType is “string”. This regular expression in ECMA-262 syntax MUST be successfully validated for all attributes that use this syntax definition. 0..1
0.FDO/NumericInterval An interval for the value (space of real numbers), that has different meanings depending on the primitive data type. If 0.FDO/PrimitiveDataType is “string”: a range of the length of the value. If 0.FDO/PrimitiveDataType is “number” or “integer”: an interval that restricts the attribute value 0..1
0.FDO/Whitelist Specifies that a specific value is allowed to instantiate this attribute. If 0.FDO/Whitelist appears multiple times, this gives an exhaustive list of allowed values to instantiate the attribute. No other values than these are permitted 0..*
0.FDO/Blacklist Specifies that a specific value is not allowed to instantiate this attribute. If 0.FDO/Blacklist appears multiple times, this gives a non-exhaustive list of forbidden values for the attribute. Other restrictions such as 0.FDO/Regex or 0.FDO/NumericRange still apply. 0..*
