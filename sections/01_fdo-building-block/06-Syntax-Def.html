Syntax Definitions

[R6-1] Syntax Definition Purpose

FDO syntax definitions (short: syntax definitions) define syntactic validation rules for attribute definitions. All syntax definitions are defined in FDO records that MUST instantiate at least 0.FDO/SyntaxDef They are referenced by attribute definitions (see ).

[R6-2] Mechanisms for Syntax Definition

The following rules can be used to define a syntax: An attribute value can be specified to
  • follow one of the primitive data types “string”, “number”, “integer” and “boolean”.
  • comply with a regular expression.
  • lie within a numeric range.
  • be equal to one element from a whitelist.
  • be not equal to all elements from a blacklist.

[R6-3] Syntax Definition are FDOs

All syntax definitions MUST themselves be represented as FDOs that instantiate the syntax definition profile. Hence, they are identified by a PID that resolves into an FDO record consisting of attributes.

[R6-4] Syntax Definition Profile

The FDO syntax definition profile defines a set of attribute definitions that MUST be instantiated in the FDO record of each syntax definition, according to the cardinalities in the attribute definitions. The FDO syntax definition profile is defined by the FDO Forum, has the PID 0.FDO/SyntaxDef and contains the list of attribute definitions shown in Table 4. It instantiates the profile definition profile and extends the root profile to ensure all syntax definitions are valid FDOs.

[R6-5] Domain-Specific Validation

The FDO record of a syntax definition MAY have additional attributes and instantiate additional profiles for any reason. For example, domain-specific validation technologies (e.g., SHACL, JSON scheme) MAY be expressed using this. However, their implementation MUST be considered optional for generic FDO clients.

[R6-6] Community Syntax Definitions

Community-specified syntax definitions MAY be implemented by communities of practice in the DTR or any other suitable open registry. Special rules apply in case syntax definitions are updated, see – .
